# SmartStay — отчёт об исправлениях

Документ описывает все изменения, внесённые в ходе аудита проекта: что было не так,
что исправлено и в каких файлах. Сгруппировано по приоритету.

> **Важно перед деплоем**
> 1. Ротировать `ANTHROPIC_API_KEY` (старый ключ засветился).
> 2. Проверить компиляцию: `python -m py_compile main.py database.py buffet.py telegram.py config.py`
> 3. Все ранее авторизованные пользователи разлогинятся (формат кук изменился) — нужно войти заново.

## 🔴 Критично — безопасность

### 1. Подделываемые куки авторизации → подпись HMAC
**Было:** привилегированные куки имели статичные значения: `admin_auth=yes`,
`manager_auth=yes`, `auth_{slug}=yes`, `owner_session={id}`, `staff_{slug}={id}`,
`hotel_slug={slug}`. Любой мог подставить такую куку запросом и получить доступ
админа / менеджера / владельца без пароля.
**Стало:** все куки подписываются HMAC-SHA256 секретом `GUEST_TOKEN_SECRET`
(по образцу уже существовавших гостевых токенов). Без серверного секрета валидную
подпись собрать нельзя.
**Файлы:** `main.py` (хелперы `_make_signed` / `_read_signed` / `_is_admin` /
`_is_manager_global` / `_verified_hotel_slug_cookie`; обновлены все места выдачи и
проверки кук).

### 2. Слабые секреты в `.env`
**Было:** `MANAGER_PASSWORD=smartstay2025`, `ADMIN_PASSWORD=admin_smartstay_2026`.
**Стало:** заменены на длинные случайные значения; добавлен отдельный
`GUEST_TOKEN_SECRET`, чтобы подпись кук не зависела от API-ключа.
**Файл:** `.env`

### 3. Stored-XSS (несколько мест)
Регистрация открытая, поэтому злоумышленник мог зарегистрировать отель и разместить
скрипт на странице в домене SmartStay (фишинг + запросы к API от имени зашедшего
админа). Исправлено экранирование контролируемых отелем и гостем данных:

- Публичная страница отеля — `name`, `page_description`, `amenities`, `ai_name`
  экранированы; `photo_url` / `booking_url` проходят валидацию (только http/https).
  **Файл:** `templates/public_html.py`
- Имя отеля на страницах чата / заселения / виджета — экранировано.
  **Файлы:** `templates/chat_html.py`, `checkin_html.py`, `widget_html.py`
- QR-страница (без авторизации): имя отеля вставлялось прямо в JS-строку
  `const name = '...'` — классический breakout. Теперь через `json.dumps` с защитой
  от `</script>`, в HTML — через экранирование. **Файл:** `main.py`
- Страница входа персонала — имя отеля экранировано. **Файл:** `main.py`
- Дашборд: имя отправителя в чате персонала вставлялось в `innerHTML` без
  экранирования; имя отеля в заголовке. **Файл:** `templates/dashboard_html.py`
- Дашборд владельца — экранирования не было вообще; добавлен `esc()`.
  **Файл:** `templates/owner_html.py`
- Поле «номер» гостя: при самостоятельном заселении не валидировалось и выводилось
  в дашборде без экранирования. Добавлена валидация номера (буквы/цифры, ≤20) +
  экранирование вывода `g.room` и `r.room`. **Файлы:** `main.py`, `dashboard_html.py`

## 🟠 Функциональные баги

### 4. Ответы персонала не доходили до гостя
**Было:** гостевой чат опрашивает `/room/{room}/new-messages`, но эндпоинт пускал
только персонал/менеджера → гость получал 401 на каждый опрос.
**Стало:** эндпоинт пускает и гостя, владеющего номером (подписанная QR-кука).
**Файл:** `main.py`

### 5. Стриминг чата терял токены и бил UTF-8
**Было:** парсер делил каждый сетевой чанк по `\n` без буфера — строка `data:`,
разорванная на границе чанка, молча терялась; многобайтный UTF-8 мог биться.
**Стало:** добавлен буфер незавершённой строки и `decode(stream:true)`; пустой ответ
ассистента больше не пишется в историю при ошибке.
**Файлы:** `templates/chat_html.py`, `widget_html.py`

### 6. Прочее
- Идемпотентность вебхука Stripe: таблица `stripe_events` + проверка по `event.id`
  (повторная доставка не обрабатывается дважды). **Файлы:** `database.py`, `main.py`
- Вебхук Telegram падал на апдейтах без сообщения (`edited_message`, null-значения).
  **Файл:** `main.py`
- Имя файла XLSX с турецкими/кириллическими символами ломало HTTP-заголовок
  (теперь `filename*=UTF-8''` с процент-кодированием). **Файл:** `main.py`
- Хрупкий `stripe.util.UNSET` в Checkout → собираем аргументы условно. **Файл:** `main.py`
- Падение парсинга `fill_percent` в анализе буфета на значениях вида `"75%"`. **Файл:** `buffet.py`
- Расхождение часовых поясов в аналитике (UTC против локального времени записи) →
  везде локальное время. **Файл:** `main.py`
- Ложные срабатывания детектора заявок (ключ `"su"` совпадал внутри «russia» и т.п.) →
  сопоставление по целым словам (regex `\b`). **Файл:** `database.py`

## 🟡 Устойчивость / валидация

- **SQLite WAL-режим** (+ `synchronous=NORMAL`, `busy_timeout=5000`) — снимает
  большинство «database is locked» под нагрузкой. **Файл:** `database.py`
- **Формат slug** при регистрации: `^[a-z0-9-]{2,40}$` (используется в URL и именах
  кук). **Файл:** `main.py`
- **Лимит истории чата** — обрезается до последних 20 реплик (защита от слива токенов).
  **Файлы:** `main.py`
- **Лимиты запросов**: welcome-эндпоинт `10/min`, демо-`/chat` снижен до `8/min`.
  **Файл:** `main.py`
- **Валидация заявки**: длина номера/сообщения/имени в ручном создании. **Файл:** `main.py`
- **Привязка отеля к владельцу** проверяет существование обеих сторон. **Файл:** `main.py`

## 🧹 Удаление легаси-демо (одно-арендный режим)

Удалён орфанный демо-кластер (не был связан с рабочим многоарендным потоком):
- Маршруты: `POST /chat`, `POST /api/login`, общий `GET /dashboard`,
  `GET /api/messages`, `POST /api/mark-read`, `GET /qr/{room}`, `GET /qrcodes`.
- Мёртвый код: хелпер `_is_manager_global`, модель `LoginRequest`, константа
  `HOTEL_INFO`, функции БД `save_message` / `get_messages`, неиспользуемые импорты.
- `config.py` больше не требует `MANAGER_PASSWORD` (служил только демо-входу);
  переменная убрана из `.env`.

Сохранены и работают: реальная форма входа менеджера `/login` (запрашивает slug +
пароль → `/api/hotel-login` → `/hotel/{slug}/dashboard`) и `/register`.
**Файлы:** `main.py`, `database.py`, `config.py`, `.env`

## 🌐 Локализация (переводы)

Было: интерфейс переведён только на 4 языка, но при «Auto» и для 23 других языков
показывался по-английски; публичная страница — только по-русски; уведомления
персоналу — смесь турецкого и русского.

- **Чат гостя:** добавлен переключатель языка (EN/RU/TR/UZ) и умный выбор —
  сохранённый выбор → язык отеля → язык браузера → английский. Больше не «всегда
  английский» при настройке «Auto». **Файл:** `templates/chat_html.py`
- **Публичная страница отеля:** переведена на EN/RU/TR/UZ, добавлены авто-определение
  языка и переключатель (раньше была только на русском). **Файл:** `templates/public_html.py`
- **Страница заселения:** тот же умный выбор языка (браузер/настройка/английский).
  **Файл:** `templates/checkin_html.py`
- **Уведомления персоналу (Telegram):** все приведены к языку отеля (EN/RU/TR/UZ) —
  заезд, новое сообщение, срочное, низкая оценка, ответ персонала, авто-выезд,
  ежедневная сводка, напоминание о выезде. Раньше срочные были по-русски, остальное —
  по-турецки. **Файлы:** `notifications.py` (новый словарь), `main.py`

- **Дашборд персонала:** воронка/графики теперь перерисовываются сразу при смене
  языка (раньше «отставали»); захардкоженные кнопки («Okundu»/«Yenile») заведены в
  переводы. Добавлены QR check-in и кнопка «📱 QR» в карточке гостя.
- **Админ-панель (Super Admin):** была целиком на турецком/русском без выбора языка —
  добавлен переключатель EN/RU/TR/UZ и переведён весь видимый интерфейс (меню,
  статистика, доход, таблица отелей, владельцы, модалки). **Файл:** `templates/admin_html.py`

Примечание: язык AI-ответов по-прежнему поддерживает все ~27 языков (определяется
по сообщению гостя) — это отдельно от языка интерфейса.

**Ещё не локализовано (можно доделать):** HTML-тела email-уведомлений (сводка,
срочное/новое сообщение) остались на турецком — email опционален и используется реже,
чем Telegram. И текстовые ошибки от бэкенда гостю (напр. «Geçersiz oda numarası») —
для них нужен проброс языка гостя на сервер.
